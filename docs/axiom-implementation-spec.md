# Axiom — Implementation Spec & Phased Build Plan
### The spec for the tool that builds code from specs.
**Author:** Ryan Colkitt | Vaspera Capital  
**Version:** 0.1 — Draft  
**Date:** April 2026  
**Purpose:** This document is the canonical implementation reference for Claude Code (or any AI coding agent) to build Axiom correctly. It defines architecture, tech stack, file structure, the `.axiom` spec format, the core regeneration loop, and a phased delivery plan where each phase proves a progressively harder use case.

---

## 0. Guiding Principles

1. **Each phase must produce a working, demoable artifact.** No phase is "infrastructure only." Every phase ends with a use case that runs end-to-end.
2. **Dogfood immediately.** Starting in Phase 2, Axiom's own utility code is written as `.axiom` specs and regenerated by Axiom itself.
3. **Prove, then generalize.** Each phase proves a specific class of use case works before the next phase broadens scope.
4. **No premature abstraction.** Build the concrete thing first. Extract the pattern only after you've built it twice.
5. **The spec format evolves with the phases.** Phase 1 supports a minimal subset. Each phase adds spec features as needed by its use case.

## 1. Tech Stack Decisions

### 1.1 Core Runtime Language: **Python 3.12+**

**Rationale:**
- Axiom's first generation target is Python. Writing the tool in the same language simplifies dogfooding — Axiom can regenerate its own modules.
- Fastest iteration speed for a solo/small founder team.
- Rich ecosystem for LLM integration (anthropic SDK, openai SDK, litellm).
- Sufficient performance for a CLI/CI tool (not a hot-path runtime).

### 1.2 CLI Framework: **Click**

- Mature, well-documented, composable.
- Clean subcommand pattern: `axiom init`, `axiom build`, `axiom verify`, `axiom cache`.

### 1.3 Spec Parser: **Custom parser using Pydantic models**

- `.axiom` files are YAML-based with custom extensions.
- Parsed into Pydantic v2 models for strict validation and type safety.
- Pydantic gives us serialization/deserialization for free (spec → IR → JSON → cache key).

### 1.4 LLM Integration: **LiteLLM (abstraction) + direct Anthropic SDK (primary)**

- LiteLLM provides the multi-model abstraction layer.
- Direct Anthropic SDK for Claude-specific features (extended thinking, tool use if needed).
- Model selection is configurable per-project and per-spec.

### 1.5 Testing / Verification: **pytest + Hypothesis**

- pytest for example-based assertion execution.
- Hypothesis for property-based invariant verification.
- Custom harness wraps both and reports in Axiom's format.

### 1.6 Package Management: **uv**

- Modern, fast Python package manager.
- Handles virtual environments, lockfiles, and dependency resolution.

### 1.7 Project Config: **`axiom.toml`**

- Project-level configuration (target language, model preferences, cache location, etc.).
- TOML for readability and ecosystem convention alignment.

## 3. The `.axiom` Spec Format

### 3.1 Format: YAML with Conventions

The `.axiom` file format is YAML. No custom syntax, no new parser to learn. The "language" is a set of YAML keys with defined semantics. This is a deliberate choice: YAML is familiar, tooling exists everywhere, and it avoids the adoption barrier of a novel syntax.

### 3.2 Spec Schema (Phase 1 — Minimal)

```yaml
# Every .axiom file must have these top-level keys:
axiom: "0.1"                      # Spec format version

metadata:
  name: string                    # Unique identifier (snake_case)
  version: string                 # Semver
  description: string             # Human-readable one-liner
  target: string                  # Generation target: "python:function" | "python:fastapi" | "python:class"
  tags: list[string]              # Optional, for organization

intent: |                         # Free-form natural language description of what this code should do.
  Multi-line string describing the purpose, behavior, and context.
  This is what the LLM reads to understand the "why."
  Be specific. Be opinionated. Describe edge cases.

interface:
  # Typed input/output contract.
  # The exact structure depends on `metadata.target`.

  # For target: "python:function"
  function_name: string
  parameters:
    - name: string
      type: string                # Python type annotation syntax
      description: string
      constraints: string         # Optional: "non-empty", "positive integer", "> 0 and < 100"
  returns:
    type: string
    description: string

  # For target: "python:fastapi"
  method: string                  # GET, POST, PUT, DELETE, PATCH
  path: string                    # e.g., "/api/users"
  request_body:                   # Optional
    fields:
      - name: string
        type: string
        constraints: string
  response:
    success:
      status: integer
      body: object                # Schema definition
    errors:
      - status: integer
        when: string              # Condition description

examples:
  # Concrete I/O pairs. These are the primary verification mechanism.
  - name: string                  # Descriptive test name
    input: object                 # Maps to interface.parameters or request_body
    expected_output: any          # Expected return value or response
    precondition: string          # Optional: setup state description
    postcondition: string         # Optional: state assertion after execution

invariants:
  # Property-based assertions that must hold for ALL valid inputs.
  # Written as natural language + optional Python expressions.
  - description: string           # Human-readable invariant
    check: string                 # Optional: Python boolean expression using `input` and `output` variables
    # If `check` is omitted, the invariant is verified by the LLM during generation
    # (it must produce code that satisfies the natural language description)
    # and spot-checked via property-based testing with generated inputs.

dependencies:
  # Other specs or hand-written modules this spec depends on.
  - name: string                  # Spec name or module path
    type: string                  # "spec" | "hand-written" | "external-package"
    interface: object             # Optional: expected interface for hand-written deps

constraints:
  # Non-functional requirements.
  performance:
    max_response_time_ms: integer # Optional
    max_memory_mb: integer        # Optional
  security:
    - string                      # e.g., "no SQL injection", "sanitize all HTML input"
  compatibility:
    python_version: string        # e.g., ">=3.12"
    packages: list[string]        # Allowed external packages, e.g., ["pydantic>=2.0", "httpx"]
```

### 3.3 Spec Schema Evolution by Phase

| Phase | Additions to Schema |
|-------|-------------------|
| **Phase 1** | `metadata`, `intent`, `interface` (function only), `examples`, `invariants` (basic) |
| **Phase 2** | `interface` (fastapi target), `constraints.performance` |
| **Phase 3** | `dependencies` (spec-to-spec), `metadata.tags` |
| **Phase 4** | Cache-related metadata (pinned model version, cache policy) |
| **Phase 5** | `dependencies` (hand-written type), escape hatch interface contracts |
| **Phase 6** | `interface` (class target), stateful specs, lifecycle hooks |

### 3.4 Example: Minimal Phase 1 Spec

```yaml
axiom: "0.1"

metadata:
  name: validate_email
  version: 1.0.0
  description: "Validates an email address and returns a normalized version or an error."
  target: "python:function"

intent: |
  Takes a string that should be an email address.
  Validates format using standard rules (must have @, valid domain, no spaces).
  If valid, returns the email lowercased and stripped of whitespace.
  If invalid, raises a ValueError with a descriptive message.

interface:
  function_name: validate_email
  parameters:
    - name: email
      type: str
      description: "The email address to validate"
      constraints: "any string"
  returns:
    type: str
    description: "The validated, normalized email address"

examples:
  - name: basic_valid_email
    input:
      email: "User@Example.com"
    expected_output: "user@example.com"

  - name: valid_with_whitespace
    input:
      email: "  hello@world.com  "
    expected_output: "hello@world.com"

  - name: missing_at_sign
    input:
      email: "notanemail"
    expected_output:
      raises: ValueError
      message_contains: "@"

  - name: empty_string
    input:
      email: ""
    expected_output:
      raises: ValueError

  - name: spaces_in_middle
    input:
      email: "user @example.com"
    expected_output:
      raises: ValueError

invariants:
  - description: "Output is always lowercase"
    check: "output == output.lower()"
  - description: "Output always contains exactly one @ symbol"
    check: "'@' in output and output.count('@') == 1"
  - description: "Output never contains leading or trailing whitespace"
    check: "output == output.strip()"
```

## 4. CLAUDE.md (Claude Code Configuration)

This file goes in the repo root and instructs Claude Code on how to work within this project.

```markdown
# CLAUDE.md — Axiom Project Instructions

## What This Project Is

Axiom is a CLI tool that compiles `.axiom` spec files into working Python code using LLMs, then verifies the generated code against the spec's assertions. The core thesis: specs are the source of truth, code is a disposable regenerated artifact.

## Tech Stack

- Python 3.12+ with uv for package management
- Click for CLI
- Pydantic v2 for spec parsing and validation
- LiteLLM + Anthropic SDK for LLM calls
- pytest + Hypothesis for verification
- YAML for spec format

## Project Structure Rules

- All CLI commands live in `src/axiom/cli/` — one file per command.
- All Pydantic models for spec IR live in `src/axiom/spec/models.py`.
- Generated code is written to `generated/` and is always gitignored.
- Cache lives in `.axiom-cache/` and is always gitignored.
- Example specs live in `specs/examples/`.
- Axiom's own dogfood specs live in `specs/self/`.

## Coding Standards

- Type hints on every function signature. No `Any` unless unavoidable.
- Docstrings on every public function (Google style).
- No classes unless they carry state. Prefer functions + Pydantic models.
- Error handling: raise specific exceptions with descriptive messages. Never bare `except`.
- Logging: use `structlog` for structured logging. No print statements.
- Tests: every module has a corresponding test file in `tests/`.

## How to Run

- `uv run axiom --help` — CLI entry point
- `uv run pytest` — run test suite
- `uv run axiom build specs/examples/validate_email.axiom` — build a spec
- `uv run axiom verify specs/examples/validate_email.axiom` — verify generated code

## Key Architectural Decisions

1. **Spec files are YAML parsed into Pydantic models.** The parser is strict — unknown keys are rejected, missing required keys fail with clear errors.
2. **The generation loop is: parse → prompt → call LLM → post-process → write file.** Each step is a separate function that can be tested independently.
3. **Verification is separate from generation.** `axiom build` generates code. `axiom verify` runs assertions. `axiom build --verify` does both. They are composable.
4. **Prompts are built programmatically from the spec IR.** No prompt templates with string interpolation. The prompt builder constructs structured messages from Pydantic models.
5. **The cache key is a hash of (spec content + target + model version + axiom version).** Any change to any input invalidates the cache.

## What NOT to Do

- Never put generated code in version control.
- Never hardcode model names — always read from axiom.toml or CLI flags.
- Never catch and silence LLM API errors — surface them clearly to the user.
- Never modify a spec file programmatically without explicit user consent.
- Never import from `generated/` in Axiom's own source code (until dogfooding phase).

## Current Phase

Check the `PHASE.md` file in repo root for what phase we're building. Only implement features scoped to the current phase. Do not jump ahead.
```

## 6. Phased Build Plan

Each phase is a self-contained deliverable. Each phase has:
- **Use case**: What it proves
- **Spec format additions**: What's new in `.axiom`
- **Implementation scope**: What code gets written
- **Done criteria**: How we know it works
- **Dogfood opportunity**: What Axiom code can be rewritten as specs

---

### Phase 1: Pure Functions
**Prove that the core loop works for the simplest possible case.**

**Use Case:** Generate a single Python function from a spec, verify it against examples and invariants.

**Spec Format:** Minimal — `metadata`, `intent`, `interface` (function target only), `examples`, `invariants` (with `check` expressions only, no NL-only invariants yet).

**Implementation Scope:**
- [ ] Project scaffolding (pyproject.toml, uv setup, directory structure)
- [ ] `axiom.toml` config loading (`src/axiom/config/settings.py`)
- [ ] Spec parser: YAML → Pydantic models for Phase 1 subset (`src/axiom/spec/`)
- [ ] Prompt builder: Spec IR → structured LLM prompt (`src/axiom/codegen/prompt_builder.py`)
- [ ] Generator: Call LLM, extract code from response (`src/axiom/codegen/generator.py`)
- [ ] Post-processor: Extract function, add imports, format with black (`src/axiom/codegen/post_processor.py`)
- [ ] Example runner: Execute I/O examples against generated function (`src/axiom/verify/example_runner.py`)
- [ ] Invariant runner: Execute `check` expressions using Hypothesis (`src/axiom/verify/property_runner.py`)
- [ ] Verification harness: Orchestrate runners, report results (`src/axiom/verify/harness.py`)
- [ ] CLI commands: `axiom init`, `axiom build <spec>`, `axiom verify <spec>`
- [ ] Example specs: `validate_email.axiom`, `slugify.axiom`, `parse_csv_row.axiom`
- [ ] Tests for parser, generator, harness

**Done Criteria:**
```bash
# This must work end-to-end:
axiom build specs/examples/validate_email.axiom
# → Calls Claude, generates validate_email.py in generated/
axiom verify specs/examples/validate_email.axiom
# → Runs 5 examples: 5/5 pass
# → Runs 3 invariants: 3/3 pass (100 Hypothesis iterations each)
# → Output: "✓ validate_email — ALL CHECKS PASSED"
```

**Dogfood Opportunity:** None yet — we need the tool to exist first.

**Demo Story:** "Here's a spec for an email validator. I run `axiom build`. It generates the code. I run `axiom verify`. All assertions pass. I change the spec, rebuild, and get new code that satisfies the new spec. I never touched the code."

---

### Phase 2: FastAPI Endpoints
**Prove it works for the most common backend building block.**

**Use Case:** Generate a complete FastAPI route handler from a spec, including request validation, response serialization, and error handling. Verify via HTTP-level assertions.

**Spec Format Additions:**
- `interface` gains `method`, `path`, `request_body`, `response` schema for `python:fastapi` target
- `constraints.performance` (response time bounds)
- `examples` gain `precondition` / `postcondition` for stateful setups

**Implementation Scope:**
- [ ] Extend spec models for FastAPI target
- [ ] New prompt builder path for endpoint generation (includes Pydantic request/response models, route decorator, error handling)
- [ ] Post-processor: Generate complete FastAPI router file (imports, models, route)
- [ ] HTTP example runner: Spin up test FastAPI app, execute requests, assert on status + body
- [ ] Performance runner: Time requests, check against `constraints.performance`
- [ ] `axiom build --target python:fastapi`
- [ ] Example specs: `create_user.axiom`, `get_user.axiom`, `webhook_handler.axiom`
- [ ] Test server harness (spins up/down FastAPI for verification)

**Done Criteria:**
```bash
axiom build specs/examples/create_user.axiom
# → Generates create_user_router.py with FastAPI route + Pydantic models
axiom verify specs/examples/create_user.axiom
# → Starts test server, sends HTTP requests, verifies responses
# → "✓ create_user — 6/6 examples pass, 4/4 invariants pass, p99 < 200ms"
```

**Dogfood Opportunity:** Write Axiom's first internal utility as a spec. Candidates:
- `specs/self/validators.axiom` — input validation functions used by the spec parser
- `specs/self/cache_key_hasher.axiom` — the cache key computation function

**Demo Story:** "I wrote a spec for a user creation endpoint. Axiom generated a FastAPI route with full Pydantic validation, error handling, and response serialization. It verified the endpoint by actually sending HTTP requests. Now I change the spec to add a new field — regenerate — and it works. The old code is gone. The new code satisfies the new spec."

### Phase 3: Multi-Spec Composition & Dependencies
**Prove that specs can compose into systems, not just isolated functions.**

**Use Case:** A system of 3–5 specs that depend on each other (e.g., a user CRUD API with separate specs for create, read, update, delete that share data models). Build and verify the whole graph.

**Spec Format Additions:**
- `dependencies` with `type: spec` — declare that this spec depends on another spec's output
- Shared type definitions (spec can define reusable types other specs reference)
- `metadata.tags` for organization

**Implementation Scope:**
- [ ] Dependency resolver: Parse spec graph, determine build order (`src/axiom/spec/resolver.py`)
- [ ] Graph build: `axiom build specs/` builds all specs in dependency order
- [ ] Shared types: A spec can export types that other specs import
- [ ] Integration verification: After building all specs, run cross-spec assertions
- [ ] `axiom build <directory>` — build all specs in a directory
- [ ] `axiom verify <directory>` — verify all specs + integration
- [ ] Example: `specs/examples/user_crud/` with 4–5 specs forming a complete CRUD API
- [ ] Dependency cycle detection and clear error reporting

**Done Criteria:**
```bash
axiom build specs/examples/user_crud/
# → Resolves dependency graph
# → Builds create_user, get_user, update_user, delete_user in order
# → Generates complete FastAPI app with all routes
axiom verify specs/examples/user_crud/
# → Runs individual spec assertions
# → Runs integration assertions (create → read → update → read → delete → read returns 404)
# → "✓ user_crud system — 4 specs, 24 examples, 12 invariants — ALL PASS"
```

**Dogfood Opportunity:** Rewrite Axiom's verification reporters as specs. The reporter module takes structured results and formats them — pure transformation, perfect spec candidate.

**Demo Story:** "Here's a complete CRUD API as 4 spec files. They share type definitions and declare dependencies. I run `axiom build` on the directory — it resolves the graph and generates the whole API. I run `axiom verify` — it tests each endpoint individually AND runs an integration test that exercises the full create-read-update-delete lifecycle."

### Phase 4: Deterministic Cache & Regeneration Stability
**Prove that regeneration is stable, cacheable, and that model drift is detectable.**

**Use Case:** Build a spec, cache the result, rebuild with same inputs (cache hit), rebuild with different model version (detect behavioral diff), rebuild with changed spec (cache invalidation + regeneration).

**Spec Format Additions:**
- Cache policy metadata (pin model version, cache TTL)
- Behavioral diff configuration

**Implementation Scope:**
- [ ] Cache store: Content-addressed storage keyed by `hash(spec + target + model + axiom_version)` (`src/axiom/cache/`)
- [ ] Cache hit detection: `axiom build` skips generation if cache is valid
- [ ] Cache inspection: `axiom cache list`, `axiom cache inspect <spec>`, `axiom cache clear`
- [ ] Behavioral diff: When model version changes, re-verify cached code before regenerating. If it still passes, keep it. If not, regenerate and diff.
- [ ] `axiom build --force` — ignore cache, always regenerate
- [ ] `axiom build --dry-run` — show what would be regenerated without doing it
- [ ] Cache stats in verification output (hit/miss/stale)

**Done Criteria:**
```bash
# First build — cache miss, generates
axiom build specs/examples/validate_email.axiom
# → "Cache MISS — generating..."
# → "✓ Generated and cached."

# Second build — cache hit, skips
axiom build specs/examples/validate_email.axiom
# → "Cache HIT — skipping generation."

# Change model version in axiom.toml
axiom build specs/examples/validate_email.axiom
# → "Cache STALE (model version changed) — re-verifying cached code..."
# → "✓ Cached code still passes verification. Keeping cache."
#    OR
# → "✗ Cached code fails verification with new model context. Regenerating..."

# Change the spec
axiom build specs/examples/validate_email.axiom
# → "Cache INVALIDATED (spec changed) — regenerating..."
```

**Dogfood Opportunity:** Most of Axiom's pure utility functions should now be spec-driven. Target: 15–20% of codebase is dogfooded.

**Demo Story:** "Watch the cache in action. First build generates fresh code. Second build is instant — cache hit. I change the model version in config — Axiom re-verifies the cached code against the spec before deciding whether to regenerate. The spec is the contract; the cache is just an optimization."

---

### Phase 5: Escape Hatches — Hand-Written Code Integration
**Prove that spec-driven and hand-written code can coexist cleanly.**

**Use Case:** A spec that depends on a hand-written performance-critical module. The hand-written module has a declared interface and invariants that Axiom verifies, but Axiom doesn't regenerate it.

**Spec Format Additions:**
- `dependencies` with `type: hand-written` — path to source file + declared interface
- `type: external-package` — third-party packages as typed dependencies
- Escape hatch interface enforcement

**Implementation Scope:**
- [ ] Hand-written module registry (`src/axiom/escape/registry.py`)
- [ ] Interface enforcement: Verify hand-written module exports match declared interface
- [ ] Invariant verification on hand-written code (run same property checks)
- [ ] Composition: Generated code can import from hand-written modules via declared interface
- [ ] `axiom verify --include-escape-hatches` — verify hand-written modules meet their contracts
- [ ] Reporting: Track ratio of spec-driven vs. hand-written code, surface in dashboard

**Done Criteria:**
```bash
# Spec declares dependency on hand-written module
axiom build specs/examples/data_pipeline.axiom
# → Generates pipeline code that imports from hand-written `fast_parser.py`
# → Interface check: fast_parser exports match declared interface ✓
axiom verify specs/examples/data_pipeline.axiom
# → Verifies generated code
# → Also verifies fast_parser invariants (processes >10k records/sec)
# → "✓ data_pipeline — 8/8 examples, 5/5 invariants, escape hatch: fast_parser VERIFIED"
```

**Dogfood Opportunity:** Axiom's LLM integration layer becomes a hand-written escape hatch (provider-specific, performance-sensitive). Its interface is declared in a spec, its invariants are verified, but its code is hand-maintained. Target: 25–30% of codebase is spec-driven, with escape hatches cleanly declared.

### Phase 6: Classes, Stateful Modules & Lifecycle
**Prove it works beyond pure functions and request handlers.**

**Use Case:** Generate a stateful Python class (e.g., a connection pool manager, a rate limiter, an in-memory cache) from a spec that defines state transitions, lifecycle hooks, and behavioral invariants.

**Spec Format Additions:**

## 7. Prompt Engineering Strategy

### 7.1 Prompt Structure (Phase 1)

The prompt sent to the LLM for code generation follows a strict structure:

```
SYSTEM:
You are a code generator for the Axiom system. Your job is to produce
a Python function that satisfies the specification below. Output ONLY
the Python code — no explanation, no markdown fences, no commentary.

The generated code must:
1. Satisfy every example (input → expected output) in the spec.
2. Satisfy every invariant for all valid inputs, not just the examples.
3. Include proper type hints matching the interface definition.
4. Include necessary imports at the top of the file.
5. Handle all error cases described in the spec.
6. Be clean, idiomatic Python. No unnecessary complexity.

USER:
## Specification: {metadata.name}
Version: {metadata.version}

## Intent
{intent}

## Interface
Function: {interface.function_name}
Parameters:
{formatted_parameters}
Returns:
{formatted_returns}

## Examples
{formatted_examples}

## Invariants
{formatted_invariants}

## Constraints
{formatted_constraints}

Generate the complete Python code for this function.
```

### 7.2 Prompt Evolution

- **Phase 1:** Single function, self-contained prompt.
- **Phase 2:** Adds FastAPI context (imports, decorators, Pydantic models).
- **Phase 3:** Adds dependency context ("this function can import from X, which has this interface").
- **Phase 5:** Adds escape hatch context ("this function must use the hand-written module Y via this interface").
- **Phase 6:** Adds class structure, state management patterns.

### 7.3 Retry Strategy

If generated code fails verification:
1. Capture which specific assertions failed.
2. Append failure information to prompt: "The previous generation failed these checks: [details]. Fix these specific issues."
3. Retry up to `max_retries` (default: 3).
4. If all retries fail, report the failures clearly and suggest spec improvements.

## 8. Verification Architecture Detail

### 8.1 Example Runner

```python
# Pseudocode for example verification
def run_example(generated_module, example):
    # Import the generated function
    func = getattr(generated_module, spec.interface.function_name)
    
    # Execute with input
    if example.expected_output.get("raises"):
        with pytest.raises(expected_exception):
            func(**example.input)
    else:
        result = func(**example.input)
        assert result == example.expected_output
```

### 8.2 Invariant Runner

```python
# Pseudocode for property-based invariant checking
from hypothesis import given, strategies as st

def run_invariant(generated_module, invariant, param_strategies):
    func = getattr(generated_module, spec.interface.function_name)
    
    @given(**param_strategies)
    def check(input_data):
        try:
            output = func(**input_data)
        except Exception:
            return  # Invariants only apply to successful executions
        
        # Evaluate the check expression
        assert eval(invariant.check, {"input": input_data, "output": output})
    
    check()  # Hypothesis runs this 100+ times with generated inputs
```

### 8.3 Strategy Generation

For Hypothesis property-based testing, Axiom needs to generate input strategies from the spec's parameter types and constraints:

| Spec Type | Hypothesis Strategy |
|-----------|-------------------|
| `str` | `st.text()` |
| `str` with "non-empty" | `st.text(min_size=1)` |
| `str` with "valid email" | `st.emails()` |
| `int` | `st.integers()` |
| `int` with "> 0" | `st.integers(min_value=1)` |
| `float` | `st.floats(allow_nan=False)` |
| `list[str]` | `st.lists(st.text())` |
| `dict` with schema | `st.fixed_dictionaries(...)` |

This mapping is maintained in `src/axiom/verify/strategies.py` and extended as the spec language grows.

## 9. Error Taxonomy & User-Facing Messages

Clear error messages are critical for adoption. Every error must tell the user what went wrong AND what to do about it.

| Error Category | Example Message |
|---------------|----------------|
| **Spec Parse Error** | `✗ specs/validate_email.axiom:14 — 'interface.function_name' is required for target 'python:function'. Add a function_name field.` |
| **Generation Failure** | `✗ LLM generation failed after 3 attempts. Last error: generated code has syntax error on line 12. Consider simplifying the intent or adding more examples.` |
| **Example Assertion Failure** | `✗ Example 'missing_at_sign' failed: expected ValueError, got 'notanemail' returned as string. Your spec says this should raise — check your intent description.` |
| **Invariant Failure** | `✗ Invariant 'output is always lowercase' failed for input email='ABC@DEF.COM' — output was 'ABC@def.com'. The generated code lowercases the domain but not the local part. Add an example covering uppercase local parts.` |
| **Dependency Missing** | `✗ Spec 'create_user' depends on 'email_validator' but no spec or module with that name was found. Create the spec or register it as a hand-written dependency.` |
| **Config Error** | `✗ axiom.toml: 'generation.default_model' is set to 'gpt-5' which is not a recognized model. Supported models: claude-sonnet-4-20250514, claude-haiku-4-5-20251001, gpt-4o, ...` |
| **API Error** | `✗ Anthropic API returned 429 (rate limited). Retrying in 30s... (attempt 2/3)` |
| **Cache Error** | `✗ Cache directory '.axiom-cache' is not writable. Check permissions or set a different path in axiom.toml.` |
